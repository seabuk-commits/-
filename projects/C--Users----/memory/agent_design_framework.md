사용자가 "에이전트 설계" 관련 요청(Claude Code 에이전트로 특정 작업을 자동화하고 싶다는 요청)을 하면, 아래 프레임워크에 따라 통합 설계서(md 파일)를 작성한다.

**Why:** 사용자가 이 프로젝트 전용 지침으로 명시적으로 제공했고, "이걸로 하자 기억해줘"라고 확정함. 매번 새로 설명하지 않아도 되도록 저장.

**How to apply:** 사용자가 자동화하려는 작업을 설명하면 항상 이 절차를 먼저 따른다. 다른 일반적인 코딩/설계 요청에는 적용하지 않는다.

## 최종 산출물: 통합 설계서 (3파트)

1. **작업 컨텍스트** — 배경/목적/범위, 입출력 정의, 제약조건, 용어 정의
2. **워크플로우 정의** — 단계별 흐름도/분기/상태전이, LLM 판단 영역 vs 코드 처리 영역 구분, 각 단계의 성공기준/검증방법/실패시 처리
3. **구현 스펙 (구조 개요만)** — 폴더 구조, CLAUDE.md 핵심 섹션 목록, 에이전트 구조(단일/서브), 스킬·스크립트 목록(역할+트리거 조건), 서브에이전트 사용 시 이름/역할/입출력/데이터 전달 방식, 주요 산출물 파일 형식

**작성 범위 제한**: CLAUDE.md, AGENT.md, 스킬 파일의 상세 "내용"은 쓰지 않는다. 구조와 역할 정의 수준까지만 (상세 내용은 구현 단계에서 작성).

## 설계 원칙

### 폴더 구조
```
/project-root
  ├── CLAUDE.md                        # 메인 에이전트 지침
  ├── /.claude
  │   ├── /skills/<skill-name>
  │   │   ├── SKILL.md
  │   │   ├── /scripts                 # 결정론적 도구
  │   │   └── /references              # (선택) 도메인 지식, API 가이드
  │   └── /agents/<subagent-name>
  │       └── AGENT.md
  ├── /output                          # 산출물
  └── /docs                            # (선택) 참고 문서
```

### 판단(에이전트) vs 처리(스크립트) 역할 분리
- 에이전트가 직접: 분류/의사결정/우선순위 판단, 품질평가/정성적 분석, 컨텍스트 기반 추론, 자연어 생성/요약
- 스크립트로 처리: 파일 I/O/데이터 파싱, 외부 API 호출, 반복 처리/집계, 정적 분석/테스트 실행

### 검증 패턴 (모든 단계에 성공기준 정의)
| 검증 유형 | 적용 대상 |
|---|---|
| 스키마 검증 | 구조화된 산출물 (필수 필드/타입) |
| 규칙 기반 | 정량 기준 있는 작업 (항목수/글자수/필수섹션) |
| LLM 자기 검증 | 정성적 판단 작업 (요약품질/톤/누락여부) |
| 사람 검토 | 고위험 최종 산출물 (외부발송/의사결정) |

### 실패 처리
| 방식 | 사용 시점 |
|---|---|
| 자동 재시도 | 단순 누락/형식오류 (최대 재시도 횟수 명시) |
| 에스컬레이션 | 판단 불확실성 높거나 기준 모호 → 사람 확인 |
| 스킵+로그 | 선택적 단계, 전체 흐름 영향 없음 → 사유 기록 |

### 에이전트 구조 선택
- 단일 에이전트(기본): 워크플로우 단순, 지침 짧을 때
- 서브에이전트 분리: 지침이 길어 항상 로드하면 비효율(필요시점 로드) 또는 독립적 작업블록이 명확히 구분되고 각각 다른 도메인지식 필요할 때
- 메인(CLAUDE.md)이 오케스트레이터, 서브에이전트 간 직접 호출 금지 → 반드시 메인 통해 조율
- AGENT.md에 역할/트리거조건/입출력/참조스킬 명시

### 데이터 전달
- 파일 기반(`/output/step1_result.json`): 데이터가 크거나 구조화됨 → 권장, 중간산출물은 파일로 저장하고 경로만 전달
- 프롬프트 인라인: 데이터가 작고 단순할 때

### 스킬 vs 서브에이전트
| 스킬 | 서브에이전트 |
|---|---|
| 도구/기능 단위(작음) | 역할/책임 단위(큼) |
| 여러 에이전트 공유 가능 | 특정 워크플로우 전용 |
| 예: file-parser, api-caller | 예: code-reviewer, report-generator |

## 진행 방식

### 1단계: 작업 파악
사용자 입력 받으면 4개 영역(목표/성공기준, 작업절차, 에이전트 조직, 기술/도구) 정보 충분한지 판단. **부족한 영역만 질문, 최대 3턴.** 모두 충분하면 질문 없이 바로 설계 착수.
- "모르겠다" → Claude가 판단하고 근거 명시
- "알아서 해줘" → 합리적 기본값 적용, 핵심 선택지만 확인
- 뻔한 질문 금지, 심층적으로 접근

### 2단계: 설계서 작성
정보 확보되면 확인 없이 자동으로 통합 설계서(md) 끝까지 작성.

### 3단계: 리뷰
초안 제시, 수정할 부분 확인.
