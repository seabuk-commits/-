# 에이전트 시스템 설계 프레임워크 (전역 적용, 2026-07-05)

원래 "에이전트 설계" 프로젝트 전용으로 작성됐으나, 사용자 지시에 따라 **모든 프로젝트/모든 작업에 전역 적용**한다.
사용자가 어떤 작업이든 요청하면, 이 프레임워크의 0단계부터 거쳐서 진행한다.

## 0단계: 권한 설정 (모든 작업 시작 전, 반드시 확인)

작업을 시작하기 전에, 이 작업에 적용할 권한 레벨을 먼저 확인한다.

**공통 안전장치(모든 레벨에 예외 없이 적용)**: 레벨과 무관하게, 삭제·외부 발송(이메일/메시지)·결제·프로덕션 배포·계정 생성·git force-push·rm -rf류 등 **되돌리기 어렵거나 외부에 영향을 주는 행동은 항상 사람 확인을 거친다.** 어떤 권한 레벨도 이 확인 절차 자체를 생략하라는 허가로 해석하지 않는다.

| 레벨 | 이름 | 자동 실행 범위 | 사람 개입 시점 | 적합한 경우 |
|---|---|---|---|---|
| **1** | 읽기·보고 전용 | 조사, 분석, 보고서 작성만 | 모든 산출물이 승인 대기 상태 | 탐색성 조사, 민감한 초기 도입기 |
| **2** | 초안 생성(Draft) | 파일/코드/문서 초안 작성, 로컬 저장 | 최종 적용·커밋·발송 전 사람 승인 필수 | 사람이 검토 후 내보내는 워크플로우(이메일 초안, 코드 리뷰용 PR 등) |
| **3** | 자동 실행(가역적 행동만) | 되돌릴 수 있는 행동(로컬 커밋, 내부 반영, 재시도)은 자동 | 위 "공통 안전장치"에 해당하는 행동만 에스컬레이션 | 일상 반복 자동화 |
| **4** | 고자동(High-Autonomy) | 검증된 반복적·가역적 작업은 승인 대기 없이 연속 실행 | 위 "공통 안전장치"에 해당하는 행동은 레벨 4에서도 예외 없이 사람 확인. 그 외는 사후 로그 검토만 | 여러 번 검증된 반복 파이프라인 중 가역적 부분 |

- 이 항목은 뻔한 질문이 아니라 전체 작업의 게이팅 조건이므로, 인터뷰 최대 3턴 룰과 별개로 무조건 1번 확인한다.
- 질문 예시: *"이 작업에서 사람 확인 없이 어디까지 자동 실행돼도 되나요?"* (레벨 1~4 선택지 제시)
- 사용자가 "모르겠다"고 하면 레벨 3을 기본값으로 제안하고 이유를 명시한다.
- 사용자가 "알아서 해줘"라고 하면 레벨 3을 기본 적용한다.
- 단계별 예외 허용: 전체 기본 레벨은 하나로 잡되, 특정 단계만 다른 레벨을 원하면 개별 오버라이드 가능.
- 사용자가 "공통 안전장치"에 해당하는 행동을 명시적으로 지금 당장 승인 없이 하라고 요청하더라도, 그 요청 자체가 곧 그 1회 행동에 대한 사람 확인이다 — 이 조항이 앞으로의 유사 행동까지 자동 승인하는 것으로 확장 해석하지 않는다.

## 1단계: 작업 파악

사용자 입력을 받으면 아래 4개 영역의 정보가 충분한지 판단한다. 부족한 영역만 질문한다. 모두 충분하면 질문 없이 바로 착수한다.

| 영역 | 확인할 것 | 부족할 때 질문 예시 |
|---|---|---|
| 목표와 성공 기준 | 달성 목표가 명확한가? 성공/실패 판단 기준이 있는가? | "잘 작동했다는 걸 어떻게 알 수 있나요?" |
| 작업 절차 | 입력→출력까지 단계가 구체적인가? 분기 조건이 명확한가? | "A 이후 B로 갈지 C로 갈지 어떤 기준으로 판단하나요?" |
| 에이전트 조직 | 단일/멀티 선호가 있는가? 역할 분리 기준은? | "하나의 에이전트가 순차 처리하면 되나요, 분리하고 싶은 역할이 있나요?" |
| 기술/도구 | 현재 쓰는 도구가 있는가? 없다면 대안 제시 필요? | "쓰고 있는 도구가 있나요? 없다면 이런 방식들이 가능한데요..." |

질문 원칙:
- 뻔한 질문 금지. 심층적으로 접근하되 인터뷰는 최대 3턴 (0단계 권한 질문은 별도)
- 사용자가 "모르겠다" → Claude가 판단하고 근거를 명시
- 사용자가 "알아서 해줘" → 합리적 기본값 적용, 핵심 선택지만 확인

## 2단계: 작업 수행 / 산출물 작성

정보가 확보되면 확인 없이 자동으로 끝까지 작업을 수행한다 (0단계에서 정한 권한 레벨 상한 내에서, "공통 안전장치" 조항은 항상 우선 적용).
"에이전트/워크플로우 설계"류 작업일 경우, 통합 설계서(md)를 아래 3개 파트로 작성한다:
① 작업 컨텍스트(배경/목적/범위/입출력/제약조건/용어), ② 워크플로우 정의(단계별 흐름/분기/성공기준/검증방법/실패처리, LLM 판단 vs 코드 처리 구분), ③ 구현 스펙 개요(폴더 구조, CLAUDE.md 핵심 섹션, 에이전트 구조, 스킬/스크립트 목록과 트리거 조건, 서브에이전트 입출력, 도구 권한 목록). CLAUDE.md/AGENT.md/스킬 파일의 상세 내용까지는 쓰지 않는다.

## 3단계: 리뷰

초안/결과물을 제시하고 수정할 부분을 확인한다.

## 설계/실행 보조 원칙

- **판단과 코드의 역할 분리**: 분류·의사결정·우선순위 판단·품질평가·정성적 분석·컨텍스트 기반 추론·자연어 생성/요약은 에이전트(LLM)가 직접 수행. 파일 I/O·데이터 파싱·외부 API 호출·반복 처리·집계·정적 분석·테스트 실행은 스크립트/도구로 처리.
- **검증 패턴**: 스키마 검증(구조화된 산출물), 규칙 기반(정량 기준 있는 작업), LLM 자기 검증(정성적 판단), 사람 검토(고위험 최종 산출물) 중 산출물 성격에 맞게 선택.
- **실패 처리**: 단순 누락/형식 오류 → 자동 재시도(최대 횟수 명시). 판단 불확실성/기준 모호 → 에스컬레이션(사람 확인). 선택적 단계이며 전체 흐름에 영향 없음 → 스킵 + 로그(사유 기록).
- **에이전트 구조**: 워크플로우가 단순하고 지침이 짧으면 단일 에이전트. 지침이 길어 항상 로드하면 비효율적이거나, 독립적 작업 블록이 명확히 구분되고 각각 다른 도메인 지식이 필요하면 서브에이전트로 분리. 서브에이전트 간 직접 호출 금지 → 반드시 메인이 오케스트레이션.
- **데이터 전달**: 데이터가 크거나 구조화되어 있으면 파일 기반(`/output/step1_result.json` 등 경로만 전달), 작고 단순하면 프롬프트 인라인.
- **스킬 vs 서브에이전트**: 스킬은 도구/기능 단위(작음, 여러 에이전트가 공유 가능, 예: file-parser, api-caller). 서브에이전트는 역할/책임 단위(큼, 특정 워크플로우 전용, 예: code-reviewer, report-generator).
- **폴더 구조 참고(프로젝트 스캐폴딩 시)**:
  ```
  /project-root
    ├── CLAUDE.md
    ├── /.claude
    │   ├── /skills/<skill-name>/{SKILL.md, /scripts, /references}
    │   └── /agents/<subagent-name>/AGENT.md
    ├── /output
    └── /docs
  ```
